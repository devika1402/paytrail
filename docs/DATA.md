# The data

paytrail runs on synthetic payment data: transactions generated by a computer
simulation. This page says what that data is, what each
field means, and what the pipeline adds on top of it, so any number in the dashboard
or a downstream report can be traced back to the raw shape it came from.

The dataset is **PaySim**, a public, openly licensed set of mobile-money transactions
(payments made from a phone-based account rather than a bank branch). It was produced
by a simulator calibrated on aggregated, anonymised transaction logs from a
mobile-money service, so the *patterns* are realistic while no row maps to a person
or account that exists. It is downloaded from Kaggle as part of setup (`make setup`), and
the source file is git-ignored because it is large.

The one fixed figure worth stating up front: the full set is **6,362,620 rows**. Every
row is one payment event.

---

## What one row is

The grain (the thing one row represents) is a single transaction event: one movement
of money at one point in simulated time, with the balances of both sides before and
after. There is no customer master table, no product catalogue, and no separate
accounts table in the source. Everything the dataset knows is on the transaction row
itself, which is why the pipeline has to *derive* accounts, dates, and segments from
these rows rather than looking them up. Those derivations are covered at the bottom of
this page.

---

## The columns

| Source column | What it means | Notes |
|---|---|---|
| `step` | When it happened, as an hour counter | 1 step is one simulated hour, counting up from the start of the simulation. There are no calendar dates in the source. |
| `type` | What kind of payment it is | One of five values, explained below. |
| `amount` | How much money moved | A non-negative amount, in the simulation's currency units. |
| `nameOrig` | Who sent the money (the originator) | An opaque account id. Its first letter encodes whether it is a customer or a merchant (see Actors). |
| `oldbalanceOrg` | Sender's balance before | |
| `newbalanceOrig` | Sender's balance after | |
| `nameDest` | Who received the money (the destination) | An opaque account id, same customer/merchant convention. |
| `oldbalanceDest` | Receiver's balance before | |
| `newbalanceDest` | Receiver's balance after | |
| `isFraud` | Was this transaction fraudulent | The ground-truth label: the simulation knows which transactions it generated as fraud. |
| `isFlaggedFraud` | Did a simple rule flag it | A rule the simulator applies to unusually large transfers. It is a crude control that catches very few of the actual fraud rows. |

(The source spells the sender's two balance columns inconsistently, `oldbalanceOrg`
but `newbalanceOrig`. The pipeline keeps the raw names in the bronze layer and
renames to consistent keys from silver onward.)

## The transaction types

`type` is the single most important field for reading the dashboard, because behaviour,
amounts, and fraud all differ sharply by type. The descriptions below are the ones the
pipeline itself carries in its transaction-type dimension.

- **CASH_IN**, customer deposits cash into an account via a merchant. Money flows in.
- **CASH_OUT**, customer withdraws cash from an account via a merchant. Money flows out.
- **TRANSFER**, funds moved between customer accounts.
- **PAYMENT**, customer pays a merchant for goods or services. Money flows out.
- **DEBIT**, funds sent from an account to a bank. Money flows out.

The amounts across these types span a very wide range (large TRANSFER and CASH_OUT
values against small DEBIT values), which is why the dashboard uses ranked, labelled
bars rather than one shared axis: on a single linear scale the small types would be
invisible.

---

## Actors: customers and merchants

Every account id is either a customer or a merchant, and the source encodes which in
the first letter of the id: `C...` is a customer, `M...` is a merchant. The pipeline
reads that first letter to label each side of a transaction as `customer` or
`merchant`, and it is why some views (customer segments, originated volume) only apply
to the customer side.

One quirk to know: merchant balances are not populated in the source. When the sender
or receiver is a merchant, its before and after balance columns are not meaningful, so
balance-based reasoning only applies to customer accounts.

---

## The fraud labels

Two columns describe fraud, and they mean different things.

- `isFraud` is the **truth**. The simulation generated certain transactions as fraud
  and labelled them. This is the flag every fraud figure in the project is built on.
- `isFlaggedFraud` is a **rule**. The simulator flags transactions its own crude rule
  considers suspicious (large single transfers). It agrees with the truth only rarely,
  which is exactly the point: a naive rule is a poor fraud control.

Fraud in this dataset is rare and concentrated: it occurs only in TRANSFER and
CASH_OUT transactions, following the simulated pattern of moving money out of an
account and then cashing it out. This is why the dashboard reports fraud as a *rate*
per type rather than a raw count, and why the overall rate is a small fraction of a
percent.

There is no fraud-detection model in this project, and that is deliberate. The dataset
carries the fraud flag, but this pipeline focuses on making a reported figure reconcile
to the source.

## Time in the dataset

Because the source only has an hourly `step` counter and no calendar dates, the pipeline
anchors step 1 to a fixed synthetic start (`2023-01-01 00:00:00`) and adds one hour per
step to produce an event timestamp. The *shape* of activity over time is accurate (it comes
straight from `step`). The specific calendar dates are a convention, chosen only to give
the steps a timeline. The derivation is in
[stg_transactions.sql](../dbt/models/silver/stg_transactions.sql), and the date
dimension is built from the same expression.

---

## Data quality: what the checks find

Standard PaySim is clean, so on the PaySim sample the pipeline's contract checks pass for
every row and nothing is quarantined (moved aside with a reason instead of discarded).
That is expected: the checks exist to catch corruption if it ever arrives
(null account ids, unparseable or negative amounts, negative balances, an unknown actor
letter), and the project proves they work by running them against deliberately corrupted
rows in unit tests. The full list of quarantine reasons and the reconciliation control
(`bronze = clean + quarantine`, with amounts tying to the penny) is in the silver
models under [dbt/models/silver/](../dbt/models/silver/).

## What the pipeline adds that the source does not have

Several fields that appear downstream, especially on the dashboard, are **derived** by
the pipeline and are not columns in the source data. Reading them as if they were raw
data would be a mistake, so they are named here.

- **Tokenised account keys.** Raw account ids are hashed (turned into a fixed
  fingerprint) so no original id flows past the silver layer. The hash is a
  pseudonym rather than anonymisation: an account id is low-entropy, so an unsalted
  hash of it can be brute-forced, and a production version would use a keyed hash
  (HMAC) or a held-back salt. The customer/merchant letter is read first, then
  discarded.
- **Event timestamp and calendar.** Derived from `step` as described above.
- **Customer segment (high / medium / low).** Not in the source. It is a band computed
  by splitting customers into three equal groups by their total originated amount, the
  volume tier a product manager slices markets by. Because the band is defined
  by volume, a chart of volume by segment is partly definitional and should be read as
  the spread of activity across tiers.
- **"Settled" framing.** The source has no settlement status. "Settled volume" is a
  reporting frame meaning the sum of transaction amounts (fraud included). The pipeline
  does not net or discard anything. This is a reporting pattern demonstrated on synthetic
  data, and it files nothing with anyone.

## Where the data is stored

The source file is stored in and read from **ADLS Gen2** (Azure Data Lake Storage
Gen2, Microsoft's cloud file storage for large data), then written untouched into the
bronze layer. Azure's role here is storage only: it stores the source file. The compute
that processes it runs on Databricks. The exact boundary of what is and is not claimed
about Azure is in [docs/AZURE.md](AZURE.md). The layered bronze, silver, and gold
model these rows flow through is explained in [docs/CONCEPTS.md](CONCEPTS.md).
